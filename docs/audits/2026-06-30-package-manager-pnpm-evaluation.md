# Package Manager Evaluation: Yarn v1 → pnpm

- **Date:** 2026-06-30
- **Type:** Research snapshot / evaluation (frozen on write)
- **Question:** Should this monorepo move off Yarn Classic, and if so, to what?
- **Short answer:** pnpm, with `nodeLinker: hoisted` for the Expo apps today. Not Bun, not Yarn Berry, not npm, and not "stay on Yarn v1."

> This is a point-in-time snapshot. Versions and ecosystem support cited here were
> verified against official docs on 2026-06-30 and will drift. If/when a decision is
> taken, record it as an ADR; do not edit this file.

| -------------------- | ----------------------------------------------------------------------------------------------------- |

> `apps/organizer` (the original Expo app, with its own `yarn.lock`) is slated for
> deletion and is **excluded from this evaluation** — no choices are made around it.

**Three repo-specific forces shape this decision:**

1. **Worktree-heavy workflow.** Work happens in `.claude/worktrees/…`. Every worktree
   is a fresh full install under Yarn v1 — hundreds of MB copied each time.
2. **One Expo / React Native app on SDK 53** (`apps/organizer-v2`). It is the most
   fragile part of the toolchain and the reason the repo carries a non-Yarn lockfile.
3. **A documented Dependabot history.** CLAUDE.md records that a stray
   `package-lock.json` once produced 159 of 299 phantom alerts (53%). Lockfile
   discipline is a known pain point here.

## 2. The options

Five realistic paths. Bun, Berry, npm, and "stay" are all rejected below; pnpm is the
recommendation.

### Option A — Stay on Yarn Classic v1 (rejected)

**Pros**

- Zero migration cost; everything works today.
- The team already knows it; CI and Vercel already consume `yarn.lock`.

**Cons**

- **Frozen software.** Yarn Classic entered maintenance mode in **January 2020**; the
  last release was **1.22.22 (March 2024)**. We're on 1.22.21 — behind even the final
  v1. It receives critical/security fixes only; all real work is on Berry 4.x.
- **Phantom dependencies.** v1's flat hoisting lets any workspace import a package it
  never declared (a sibling hoisted it to root `node_modules`). With shared
  `packages/api|db|transactional` consumed by `apps/web`, this is exactly the setup
  where undeclared imports work locally and break on a clean publish/build.
- **No worktree disk relief.** Each worktree gets a full copied tree.
- **The lockfile split persists.** Nothing forces consolidation.

**Verdict:** Building on frozen tooling while the known pain points (worktree cost,
phantom deps, lockfile drift) all go unaddressed. Reject.

### Option B — Upgrade to Yarn Berry (v4) (rejected)

**Pros**

- Actively maintained; Expo lists Yarn Berry as first-class supported for monorepos.
- Modern lockfile, `resolutions`, constraints, plugin system.

**Cons**

- **Its headline feature (PnP) is unusable here.** React Native / Metro cannot run
  under Plug'n'Play, so you'd immediately set `nodeLinker: node-modules` — discarding
  the one thing Berry does that v1 doesn't.
- **High migration churn for low gain.** New `.yarnrc.yml`, in-repo Yarn binary, plugin
  model, lockfile format change — to land on "a more complex v1."
- **Weak worktree payoff.** With the node-modules linker, Berry still materializes a
  full tree per worktree. Its global cache dedupes downloaded _archives_, not the
  installed tree — so the worktree-disk win (our #1 motivation) mostly evaporates.

**Verdict:** Maximum churn, minimal benefit _because_ we have Expo. Reject.

### Option C — npm (rejected)

**Pros**

- Ships with Node; nothing to install. Has workspaces and `overrides` (since npm 7+).
- Expo supports npm.

**Cons**

- **Same phantom-dependency model as Yarn v1** — npm workspaces hoist to root
  `node_modules` by default, so undeclared imports work in-repo and break standalone.
  Switching from v1 to npm trades one flat-hoister for another and fixes none of the
  structural problems.
- **No content-addressable store / hardlinks** → no worktree disk win.
- **Directly re-opens the Dependabot wound.** The 159-alert incident was _caused by_ a
  `package-lock.json`. Making npm the primary manager institutionalizes the exact
  lockfile we've been fighting.

**Verdict:** A lateral move that re-introduces a documented pain point. Reject.

### Option D — Bun (rejected, with a concrete blocker)

**Pros**

- Fastest installer of the group; text `bun.lock`; global cache with hardlinks.
- Expo's docs _list_ Bun as a first-class supported monorepo package manager.

**Cons**

- **Live, unresolved Metro bug.** [expo/expo#41995](https://github.com/expo/expo/issues/41995)
  (opened 2026-01-07, still open): Metro **cannot resolve transitive dependencies**
  under Bun because Bun stores them in `node_modules/.bun/` instead of hoisting. Affects
  Bun 1.x + Expo 54 + RN 0.81. The only workaround is manually promoting every
  transitive dep (e.g. `promise`, which RN pulls in) to a direct dependency — untenable.
- This lands the blocker squarely on our **most fragile subsystem** (`apps/organizer-v2`).
- Tooling maturity (audit, Dependabot) for Bun lockfiles still trails pnpm/npm.

**Verdict:** Right tool for a greenfield server; wrong place to absorb an open RN
monorepo bug. Reject for now — revisit if #41995 closes.

### Option E — pnpm (**recommended**)

**Pros**

- **Worktree disk + speed win (unique to pnpm here).** Files live once in a global
  content-addressable store (`~/.pnpm-store`) and are **hardlinked** into each
  `node_modules` — "consuming no additional disk space." A second worktree's install is
  near-instant and near-free. This directly pays off our #1 force.
  _(Caveat: the store must be on the same filesystem as the worktrees, or pnpm copies
  instead of linking. All worktrees live under the repo on one disk, so this holds.)_
- **Strict dependency isolation where it counts.** pnpm's symlinked layout means a
  package can import only what it declares — killing phantom dependencies in `apps/web`
  and `packages/*`. This is the structural fix npm and Yarn v1 can't offer.
- **Officially supported Expo path, version-gated and future-proof:**
  - **SDK 53 (today):** Expo recommends disabling isolated deps via
    `nodeLinker: hoisted`. **`nodeLinker` is workspace-global** (it cannot be set
    per-package), so this puts the _entire_ repo — web, packages, and
    `apps/organizer-v2` — in a flat npm/Yarn-style `node_modules`. Net: **no strict
    phantom-dep isolation anywhere on SDK 53.** The disk/speed and lockfile/catalog
    wins still land immediately — they come from the content-addressable store and
    are independent of the linker mode.
  - **SDK 54+:** Expo supports the isolated (default, strict) linker directly (modern
    Metro added Global Virtual Store support for symlinked stores). Moving
    `apps/organizer-v2` to SDK 54 lets the whole workspace drop `nodeLinker: hoisted`
    and gain strict isolation everywhere — no manager change.
  - **Unofficial middle path:** keep the isolated linker now and add
    `public-hoist-pattern` entries for `*expo*` / `*react-native*` / `metro*` to lift
    just those to root `node_modules` while the rest stays strict. Community-supported,
    not Expo-blessed for SDK 53 — higher risk; validate native builds carefully.
- **Catalogs fix the React skew.** pnpm **catalogs** (`pnpm-workspace.yaml`) define a
  version once and reference it as `catalog:` across packages. Our React `19.2.1`
  (web) vs `19.0.0` (Expo) drift becomes a single source of truth; upgrades touch one
  line and stop causing merge conflicts in `package.json` files.
- First-class Vercel support; text lockfile with mature Dependabot/audit coverage —
  the path _out_ of the lockfile-drift problem, not back into it.

**Cons / trade-offs (stated honestly)**

- **No strict isolation on SDK 53.** `nodeLinker` is workspace-global, and Expo SDK 53
  needs `hoisted`, so the _whole_ repo runs flat until `apps/organizer-v2` reaches
  SDK 54. The phantom-dep win is deferred (the disk/speed/lockfile/catalog wins are
  not — they land immediately).
- **Hoist patterns are required for RN.** Need `public-hoist-pattern` entries for
  `*expo*`, `*react-native*`, `@react-native/*`, `metro*`, or native module resolution
  breaks. This is documented, not novel.
- **Occasional `--force`.** The global store can rarely need
  `pnpm install --force` after odd states.
- **Migration is real work** (Section 4) — a one-time cost, not ongoing.

**Verdict:** The only option whose hardlinked store pays off the worktree workflow, with
an officially-documented Expo escape hatch today and a clean strict-mode upgrade path at
SDK 54. **Recommended.**

## 4. Migration sketch (if pnpm is chosen)

Not a decision — a scoping aid.

1. **Pin the manager.** Add `"packageManager": "pnpm@<latest>"` to root `package.json`;
   keep Node 24. Enable via Corepack.
2. **Root `pnpm-workspace.yaml`:**
   ```yaml
   packages:
     - 'apps/*'
     - 'packages/*'
   # SDK 53: Expo needs the flat layout. nodeLinker is WORKSPACE-GLOBAL, so this
   # hoists the entire repo (publicHoistPattern is moot under the hoisted linker).
   # At SDK 54: delete this line to get the isolated (strict) linker everywhere.
   nodeLinker: hoisted
   catalog:
     react: 19.2.1 # unify the web/Expo skew here
     # add other shared deps over time
   ```
   Then reference `"react": "catalog:"` in each workspace `package.json`.
3. **Convert lockfiles.** `pnpm import` from the existing `yarn.lock` to seed
   `pnpm-lock.yaml`; delete the root `yarn.lock` and
   `apps/organizer-v2/package-lock.json`. One lockfile, one manager.
4. **Scripts.** Replace `yarn --cwd` / `yarn workspace X` invocations in root
   `package.json` with `pnpm --filter`. Update the `typecheck` script accordingly.
5. **Vercel.** Set install to `pnpm install` (auto-detected from `pnpm-lock.yaml`);
   confirm `apps/web` build.
6. **CI / husky.** Ensure lint-staged + prettier pre-commit still runs under pnpm.
7. **Verify Expo.** `pnpm install`, then `npx expo start --clear` in `apps/organizer-v2`;
   confirm native module + transitive resolution.
8. **Later (SDK 54):** drop `nodeLinker: hoisted` and adopt isolated installs repo-wide.

**Rollback:** the change is contained to lockfiles + config + script invocations; revert
the branch to return to Yarn v1.

## 6. Empirical benchmark (2026-06-30)

Run on the dev machine, scoped to `apps/web` + `packages/*` (Expo app excluded), in
throwaway detached worktrees. pnpm 10.6.1 / yarn 1.22.21 / Node 24. macOS APFS.
Re-run on a cleaned disk (~68–75 GB free, both caches empty) so timing is now reliable
and the cold comparison is fair (both managers started from an empty cache/store).

| Test                                                       | Yarn v1     | pnpm 10                        | pnpm advantage      |
| ---------------------------------------------------------- | ----------- | ------------------------------ | ------------------- |
| Cold install (empty cache/store)                           | 169.6s      | 54.1s                          | **~3.1× faster**    |
| Warm install (cache/store populated, fresh `node_modules`) | 22.0s       | 16.1s                          | **~1.4× faster**    |
| **2nd-worktree true disk cost** (`df` delta)               | **1946 MB** | **132 MB**                     | **~15× less**¹      |
| `web` typecheck                                            | passes      | fails — 3 phantom deps²        | catches latent bugs |
| `node_modules` size                                        | 1.9 GB      | 1.7 GB apparent / 1.5 GB store | —                   |

1. The pnpm per-worktree figure varies run to run (80–132 MB observed) depending on how
   many files clone vs copy; the order of magnitude (~15–24× less than yarn) is stable.
2. **Key correctness finding.** pnpm's isolated linker caught three dependencies
   imported in `apps/web` source but **not declared** in `apps/web/package.json`
   (`@ant-design/icons`, `@types/google.maps`, `@tanstack/table-core`). The identical
   `tsc --noEmit` passes under Yarn (flat hoisting masks them) and fails under pnpm.
   These are latent bugs today, independent of any migration. Fix by declaring them.

**Methodology notes that bit us (record for next time):**

- On APFS, pnpm defaults to `clonefile` (copy-on-write), so **`du` overcounts** — it
  reported 3.4 GB across two worktrees that truly shared blocks. Use `df` free-space
  deltas to measure real disk, not `du`.
- **pnpm 10 changed the default:** workspace packages are only auto-linked via the
  `workspace:` protocol. A plain version range made pnpm try to fetch private
  `@troptix/api` from npm. Migration must set `link-workspace-packages=true` or rewrite
  internal deps to `workspace:*`.

**Takeaway:** pnpm won on every axis measured — ~3.1× faster cold install, ~1.4× faster
warm install, ~15× less disk per worktree, and it surfaced real phantom-dependency bugs
yarn's hoisting was hiding. The earlier "speed unsettled" caveat is resolved: on a clean
disk the speed advantage is clear, not noise.
