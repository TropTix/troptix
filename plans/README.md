## Not planned this round (real findings, deliberately deferred)

- **F9 — `CheckoutSheet` ad-hoc state machine** — **now planned as 017** (reducer + discriminated union, behind the 016 test net).
- **F10 — large-order fulfillment.** `materializeOrder` builds one `Tickets` row per unit inside the confirm/settle transaction while holding the `FOR UPDATE` lock — a 5,000-ticket order can exceed the webhook timeout. L effort; the next scalability bottleneck after 015.
- **F11 — RELEASED fulfillment gap.** `getCheckoutState` returns `expired` for RELEASED before the paid-session sync check, and `release()` never expires the Stripe Session → a payment on a released hold shows "expired" until the webhook recovers. S effort, MED confidence (edge-triggered; the webhook path still fulfills correctly).
- **F12 — test coverage cliff** at the HTTP edge (webhook + crons) and the client UI. **The client half is now planned as 016**; 014 chips at the webhook; the cron-route tests remain.
- **F13 — ADR 0021 violation** in `PaymentStep.tsx` `OrderSummary` (`toLocaleDateString/TimeString` instead of the venue-local formatter). S effort.
- **F14 — `capacity` NULL divergence.** Reads fall back to `quantity`; `holdInventoryInTx` treats NULL capacity as 0 availability. Latent (backfill-dependent). S effort.
- **F15 — cleanup bundle.** Dead `confirm()`/`expire()` exports; order/receipt pages read Prisma directly (float↔cents fallback in UI); money formatting reimplemented ~8×; stale `_shared/fees.ts` comment referencing a deleted file. S–M effort.

## Findings considered and rejected (do not re-audit)

- **Money math float/cents disagreement** — verified consistent across reads, `deriveReserveItems`, and Stripe line items; `priceCents ?? round(price*100)` used uniformly. Not a bug.
- **Webhook idempotency weak** — false: `ProcessedStripeEvent` is real and used, layered on `settle`'s `FOR UPDATE` row lock and the `refund-<id>` idempotency key. Solid.
- **`beginPayment` partial failure strands the buyer** — false: the fixed `checkout-<reservationId>` idempotency key makes Stripe replay the same Session on retry; the stale-session retry key avoids replaying a dead Session.
- **Reservation service leaks Stripe** — false: `reservations.ts` imports no Stripe; all Stripe lives in `payments.ts` (ADR 0018 honored).
- **Pay-button double-submit** — guarded by `submitting || !checkout.canConfirm` plus Stripe's own `confirm()` guard.
- **Gated-tier bypass** — false: the reservation path filters to public tiers and rejects any non-public tier. If anything the gate is over-strict (see direction option 1).
- **Fee duplication across the app** — the `/e/` flow and ticket forms already single-source `calculateFeesCents`; the stale `fees.ts` comment claiming a live legacy duplicate is wrong (the file it names is deleted) — folded into F15.
